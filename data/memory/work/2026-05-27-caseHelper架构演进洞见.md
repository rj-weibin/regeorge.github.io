# caseHelper 架构演进洞见

> 从脚本仓库到 AI Native 服务编排的演进复盘，提取可复用的工程认知。

---

## 核心认知

> Phase 0 门禁是整个项目投入产出比最高的设计。不做状态矩阵分析就批量创建，错误会被 N 倍放大。

> 这个项目最有价值的产出不是代码，而是流程认知的固化。

> 先穷举状态组合，再创建公共步骤，最后批量生成业务用例——顺序反了，成本翻倍。

---

## 五层 AI Native 分层

| 层 | 职责 | 设计约束 |
|----|------|---------|
| Skill | 工程规范/SOP/Agent 协议 | 只放纯文本规则，不含流程不含代码 |
| Service | 流程编排/状态流转 | 组合 Capability，向上暴露完整业务流程 |
| Capability | 原子能力/API/解析/转换 | 只负责"具体能做什么"，不负责阶段推进 |
| Workspace | 运行时状态/阶段产物 | 不作为长期规则仓库 |
| Knowledge | 已固化的长期规则 | 不存放一次性实验噪音 |

关键洞见：Agent 先按 Skill 获取工作方法，而不是先钻进代码细节。Service 负责把多步动作串成稳定流程，减少 prompt 级别临场拼接。

---

## Skill 演化论

> Skill 不再默认承载原子能力、复杂 workflow、低层 API SDK、通用 utils。它的核心职责变为：定义 Agent 如何思考、任务如何推进、什么时候允许继续/什么时候必须阻断、结果如何沉淀。

早期 Skill 把 SOP、workflow、脚本、API、规则、prompt 都塞在一起。短期能跑，但规模一大就会：
1. Skill 变胖，SKILL.md 逐渐变成半个平台说明书
2. 原子能力无法横向复用，开始复制粘贴
3. 流程与能力耦合，改一步流程就牵动整包实现

---

## 做减法阶段的判断标准

### 不再做的事（已验证无效）
- 不自动化 Phase 0 分析——核心价值是人的判断
- 不扩展 agent_service——Copilot Chat + 本地脚本已足够
- 不做 skills 发现/推荐/元数据系统——使用者已经知道要调哪个
- 不追求知识库双向同步——单向 "平台→本地" 已够用
- 不建单元测试框架——审计规则比 mock 测试有效
- 不拆 platform_client——工具项目中内聚的大类优于过度抽象

### 继续打磨的方向
- case-debugger 的全局公共用例识别（修 bug）
- 审计规则配置从 Phase 0 产物自动生成
- 配置源收敛到单一 ServiceConfig
- sys.path 注入治理（editable install）

---

## 可复用模式

1. **门禁制**：批量操作前的人工阻断，把审计当 CI lint 而不是事后 hotfix
2. **统一契约**：所有 service 返回 ServiceResult，不 throw 异常给调用方
3. **知识版本化**：将平台数据拉到本地 JSON 做 git 版本化，可 diff 可审计
4. **错误码驱动修复路径**：CONFIG_MISSING 修环境、API_CONTRACT_MISMATCH 修适配层、INPUT_VALIDATION_ERROR 修调用方
5. **探针即弃**：实验放 playground，结论必须上浮到 knowledge 或 meta.json，然后删除

---

## 一句话总结

> 项目已经过了"做加法"的阶段，进入了"做减法和加固"的阶段。下一阶段的重心放在打磨已有支柱的可靠性上，而不是继续扩展新能力。
