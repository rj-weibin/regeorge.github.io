一个人的自动化测试，怎么活下来

不是什么方法论，就是这几个月摸索出来的一套干活的办法。
从"拿到一堆手工用例"到"平台上跑通"，中间踩了不少坑，也攒了些心得。

😤 痛点
💡 核心思路

🪤 踩坑记录

先说说痛在哪

这些不是假设出来的，是每天干活时真实遇到的。

📋

手工用例多得看不完

一个需求下来四五十条手工用例，每条还有好几个步骤。光"看懂这条在测什么"就得花很长时间，更别说想清楚怎么自动化了。

🔍

平台接口没文档，全靠猜

自动化平台的 API 没有完整文档。method 字段到底传字符串还是数字？更新用 PUT 还是 POST？body 要不要自己序列化？全得一个一个试出来。

🔀

"设计"和"实现"永远在打架

你在 review 文档里写的是"调用 /accountFund 接口"，到了平台上建步骤时手一抖写成了 /accountRefund。然后 debug 半天不知道哪错了。流程图、review 页面、平台步骤，三个地方说的不是一回事。

💀

环境数据污染，今天跑过明天挂

上午调通了，下午就挂了。不是代码改了，是测试环境里别人的数据把你的用例干扰了。"周期到期查询应该返回 0 条"——但环境里正好有一条别人留下的到期账户。

🧩

公共用例不可控

平台上有别人写的"公共用例"，你引用了它。结果它内部的某个步骤在你的环境下跑不通（模板不存在、套餐已下线），你的用例就跟着挂了。改又不敢改——怕影响别人。

🏃

一个人要干完所有事

从"理解需求"→"拆用例"→"想自动化方案"→"写步骤"→"建到平台"→"调试"→"回填结果"→"更新文档"，全是一个人。没有专职的测试架构师帮你设计，也没有专门的工具链给你用。

一句话总结：不是技术不行，是中间过程太多、太碎，每个环节都可能出错，而且错了很难发现。

核心思路：不要一步到位

最大的教训就是——别想着"AI 看完需求就直接输出能跑的自动化脚本"，那是幻觉。

❌ 以前的做法

  • 拿到手工用例，直接在平台上建步骤
  • 凭记忆和直觉决定先做哪条
  • 建完就跑，跑不过就改，改了再跑
  • 文档？等做完了再补吧（然后永远没补）
  • 每次做新需求都从零开始

✓ 现在的做法

  • 先生成 review 页面，逐条过一遍
  • 人工审批通过后才开始建平台步骤
  • 先做 1-2 条代表案例，跑通再批量
  • 文档和实现同步更新，不允许偏差
  • 每次经验沉淀为"模式"，下次直接复用

四步闭环

🧠

读手工用例 → 生成 review.html（流程图 + 自动化步骤 + 断言）

👁️

逐条看：这条该不该自动化？接口对不对？步骤顺序合理吗？
↓
🏗️
Phase 2 · 先样本、再批量
挑 1-2 条代表用例，在平台上建步骤、调试通过，然后复制模式批量建
↓
🔄
Phase 3 · 回填 + 沉淀
批量调试 → 回填平台 ID → 更新 review.html → 踩坑经验沉淀为模式卡片
关键约束：三段对齐
流程图说的接口 = review 页面写的步骤 = 平台上建的真实步骤。
任何一处改了，另外两处必须同步。发现偏差时，先回头修 review.html，再改平台。

人和 AI 怎么分工

🤖

AI 擅长的

大量手工用例的快速理解和归类、生成标准化 review 页面、根据模式批量建步骤、自动回填 ID 和状态、格式化输出。

🧑

人必须做的

审核 AI 拆出来的方案对不对（接口有没有搞错、流程顺序对不对）、确认环境是否可用、决定哪些用例保留手工、抓包确认真实接口参数。

🤝

必须一起做的

样本验证——AI 建出来第一条，人跑一下看对不对。对了再批量。不对就纠偏，AI 修正后再试。这个循环是整个流程的核心。

警惕"AI 幻觉"：AI 不知道平台接口的真实行为。它可能自信地写出一个根本不存在的 endpoint。所以第一条必须人工验证，验证通过后的模式才可以批量复用。

几个真实故事

不是教科书案例，是最近几个月真实遇到的事。

账户管理 · 43 条用例

拿到 43 条手工用例，差点放弃

需求下来了，飞书表里 43 条手工用例。光看标题就头大——"单个添加可透支账户"、"批量添加账户重名"、"删除已关联用户的账户"……

用 AI 先过了一遍，生成了 review.html。打开一看，立刻发现 5 条用例是纯 UI 展示类（"列表字段包含XX"），这种没法接口自动化，直接标记保留手工。又发现 8 条用例依赖"造欠费/透支状态"，而我们还没有稳定的计费触发公共步骤，这些也先标记为"待确认"。

最终实际需要建的只有 30 条。如果不先过 review，我可能会试图给 43 条都建自动化，浪费大量时间在不可能跑通的用例上。

费用管理 · 缴费链路

"调试通过"和"真正通过"是两回事

费用管理的缴费用例，AI 建好步骤后在 env=123 上调试，8 条全过了。信心满满。

然后跑剩下的用例——6 条"周期到期查询"全挂了。查了半天，原因是：之前跑通的缴费用例创建了测试账户，这些账户恰好在"周期到期"的查询范围内。我的用例期望查到 0 条结果，但因为前面的测试残留数据，实际查到了 1 条。

教训：每条用例必须有独立的清理步骤，不能假设环境是干净的。

白名单 · SOAP 接口

接口文档说的和实际的不一样

白名单功能有个第三方 SOAP 接口 bindOperatorV2。平台上有人写过公共用例引用了它，但 body 是空的 {}。

AI 照着抄，当然跑不通。后来自己抓包才发现：SOAP 请求需要完整的 XML 信封，header 必须带 Content-Type: text/xml 和 Authorization: Basic dGhpcmQ6MTIz。平台上的"公共用例"根本没写这些。

最终把公共用例引用全部展开成内联 HTTP 步骤，手工补全了 XML body 和 header。不要盲信"公共"二字。

基本费计费 · RADIUS 协议

断言值到底是什么意思？

RADIUS 认证用例的断言写的是 statusCode=2，一开始以为是 HTTP 200 类似的东西。debug 半天搞不懂为什么 "statusCode=2" 就代表成功。

后来查了 RADIUS RFC 才知道：statusCode=2 是 Access-Accept（认证通过），statusCode=5 是 Accounting-Response（记账确认）。这根本不是 HTTP 状态码，是 RADIUS 协议层的语义。

把这个知识记下来变成模式卡片后，后面所有 RADIUS 用例的断言都不用再查了。踩过的坑一定要沉淀。

一路做出来的工具

不是一开始就设计好的，是遇到问题就做一个，做着做着就成了工具链。

📄

review.html 生成器

输入手工用例数据，输出可交互的审核页面。支持分类筛选、快捷键审批、一键导出结果、自动同步审核状态。最近刚做了标准化模板，新需求几行代码就能生成。

🔗

Platform Client

封装了自动化平台的所有 API（建用例、建步骤、注入变量、触发调试、读日志）。所有"平台接口到底怎么调"的坑都埋在这一层里，上层不用操心。

🐛

批量调试 + 日志解析

一键对一批用例触发 debug，自动解析每一步的 PASS/FAIL/SKIP，输出哪些用例挂了、挂在第几步、实际返回了什么。不用再一条条点开平台去看了。

📡

Station 本地仪表盘

本地起的一个 Web 服务，汇总所有需求的自动化覆盖率、用例映射、审核状态。打开浏览器就能看"这个需求还有几条没建、几条调试没通过"。

📥

飞书/CSV 批量回填

自动化用例在平台上建好后，把平台 ID 回填到飞书表和 CSV。之前全是手动复制粘贴，现在一个脚本搞定。

🧠

模式沉淀飞轮

每次踩坑的经验自动沉淀为知识卡片——接口规范、断言策略、变量命名约定、清理模式。下次遇到同类问题，直接调出来用，不用再踩一遍。

踩坑备忘录

这些坑不踩不会知道，踩了才能记住。

千万不要跳过"样本验证"

之前图快，AI 生成了 20 条用例的步骤，直接批量建到平台上。结果 18 条的接口路径是错的——AI 把 /accountFeeList 和 /accountList 搞混了。花了一下午才全部修过来。现在的规矩是：先做 1 条，调通了再批量。

body 序列化只能做一次

调了两个小时才发现，平台返回 422 是因为 body 被序列化了两次。我在调用代码里写了 json.dumps()，Platform Client 内部又做了一次。结果发出去的是双重转义的 JSON 字符串。现在统一约定：调用方传 dict，序列化只在 Client 内部做。

平台更新不能用 PUT

直觉告诉你"更新应该用 PUT"，但这个平台的正确做法是 GET → 修改 → POST。用 PUT 要么 404 要么静默失败。这种事没有文档可查，只能靠试。试出来的结论一定要记下来。

变量名拼错，debug 不会告诉你

用例 body 里写了 ${userUuid}，但上游步骤提取出来的变量名其实是 ${userinfoUuid4opt}。运行时平台不会报"变量未定义"，而是原样把 ${userUuid} 这个字面量发出去，接口收到一个字符串 ${userUuid} 当然返回错误。遇到 debug 失败，第一件事看日志里的实际 body 有没有未替换的 ${}。

负向用例必须设 exception=0

正向用例断言 code==200，失败了就中止，合理。但负向用例断言 code!=200，如果 exception 还是默认的 1（中止），那"断言成功"（确实 !=200）之后用例直接中止了，后面验证"系统状态没变"的步骤根本跑不到。负向用例的操作步骤必须设 exception=0。

别把"环境问题"和"代码问题"搞混

用例跑失败了，先别急着改步骤。看一下 debug 日志：变量替换正常（没有 ${} 残留）、接口路径没错、body 格式正确——那大概率是测试数据被别人的操作干扰了。这种时候该做的是清数据或换账号，不是改代码。区分"数据污染"和"代码缺陷"能省下大量时间。

SKIP 不一定是错

第一次看到 debug 日志里有步骤状态 res=3 (SKIP)，以为用例出 bug 了。后来才知道：公共用例里有些步骤带条件判断（比如"只在 UNC 环境执行"），在 SAM 环境下这些步骤会自动跳过，属于正常行为。先搞清楚日志里每个状态码的含义再下结论。
